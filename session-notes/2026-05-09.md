## Headline outcomes

1. **v3a is live as the launch artifact at
   [`huggingface.co/lerugray/hammerstein-7b-lora`](https://huggingface.co/lerugray/hammerstein-7b-lora).**
   Mixed-mode training (1494 strategic + 214 off-domain pairs, 12.5%
   off-domain ratio) eliminates the catastrophic-forgetting
   regression that v1's model card had flagged as a known limitation.
   Three independent measurements all favor v3a:
   - Strategic raw markers: 5.80 vs v1's 5.60 (+0.20)
   - OOD leakage: 0.00 vs v1's 2.80 (catastrophic forgetting fully
     suppressed; adapter no longer framework-ifies haikus, recipes,
     factual queries)
   - Blind LLM judge head-to-head (qwen3.6-plus, n=40): v3a wins
     27/40 (67.5%), well above the 55% conventional pairwise
     significance threshold

   Both the v3a adapter and a fresh Q4_K_M GGUF (`hammerstein-7b-lora.Q4_K_M.gguf`,
   4.7 GB) are on HF. Old v1 artifacts removed. Modelfile updated
   to point at v3a GGUF. `ollama run hf.co/lerugray/hammerstein-7b-lora:Q4_K_M`
   serves v3a.

2. **v2 ran two parallel single-variable experiments before
   committing to v3a.** The brief proposed a combined v2 (data scale
   + teacher swap + epoch bump). Hammerstein audit
   ([qwen3.6-plus, $0.0078](#)) flagged this as a confounded design
   and endorsed isolating each variable. So v2a (data 308 → 1494,
   same qwen3.6-plus teacher) and v2b (DeepSeek v4-pro teacher,
   same dataset size) ran in parallel on Mac. Both completed in
   ~70 min wall-clock. Neither was a clean launch swap:
   - **v2a:** marginally better strategic, materially worse OOD
   - **v2b:** worse strategic (DeepSeek register pulled away from
     Hammerstein voice — exactly Hammerstein's pre-flight "register
     mismatch" failure prediction), better OOD

   The audit's "isolate variables" discipline was the load-bearing
   call. A confounded combined-variable v2 would not have surfaced
   these as separate effects, and v3a's design (v2a data + the
   off-domain mitigation v2b implied) would not have been visible.

3. **v3a was named in v1's model card a day earlier as deferred
   work.** v1 documented "Mitigation if we re-train: mix 10–20%
   off-domain instruct data... standard practice for catastrophic-
   forgetting suppression. Adds maybe $1 of data-gen cost, ~10 min
   more training. Deferred for now and flagged as a known limitation."
   v3a executed exactly that: 12.5% off-domain via qwen3-coder-flash
   (no system prompt, anti-leakage filter), $0.07 OpenRouter spend,
   17 min train. The methodology arc (v1's deferred mitigation →
   v2's variable isolation → v3a's clean fix) is itself a credibility
   signal worth more than the marginal numbers.

4. **Eval set expanded from 4 → 30 OOD prompts.** The original
   forgetting-check set was a 4-prompt minimal falsification set,
   too noisy to discriminate v2 variants (v1's reported 0.312
   leakage was lucky on those 4; the n=30 set revealed v1 actually
   leaks at 0.500 capped / 2.80 raw markers). The expansion spans
   6 shape categories (creative, factual, technical-explanatory,
   instructional, conversational, math/code) — see
   [`tools/distill/eval.py:64`](../tools/distill/eval.py).

5. **README + HAMMERSTEIN-7B.md + HF model card all rewritten
   for v3a.** Cost arc table updated (v1 + v2 + v3a broken out,
   total ~$34). Status table now shows three distillation rows
   (v1 / v2 / v3a). Reproducibility section points at
   `synthetic-v3a-2026-05-09.jsonl` and the new
   `gen_offdomain.py`.

## RunPod adventures (the GGUF saga)

The v3a adapter pushed cleanly to HF via `huggingface_hub.HfApi.upload_folder`
through `tools/distill/hf_push.py`. The GGUF conversion took five
attempts across two pods:

1. **Attempt 1:** Reused training pod template, ran `convert_gguf.py`
   (Unsloth's `model.push_to_hub_gguf()` wrapper). **Failed:** "No
   space left on device" — the 30 GB container disk filled during
   the merge step.
2. **Attempt 2:** Cleared pip cache + HF blob cache, re-ran. **Failed:**
   Unsloth's wrapper called `apt-get install cmake libcurl4-openssl-dev
   libssl-dev -y` with an interactive prompt, which broke under nohup
   with "EOF when reading a line."
3. **Attempt 3:** Pre-installed the apt packages with `DEBIAN_FRONTEND=noninteractive`,
   re-ran. **Failed:** Unsloth's `unsloth_convert_hf_to_gguf.py` wrapper
   called llama.cpp's converter with `capture_output=True`, hiding the
   real error. The merged model dir was deleted by Unsloth's cleanup
   on failure, masking the actual cause.
4. **Attempt 4:** Bypassed Unsloth entirely. Wrote a manual pipeline
   shell script (peft merge → llama.cpp convert → llama-quantize → HF
   upload). Spun up a fresh pod with **80 GB disk** to avoid the disk
   issue. **Failed twice:** (a) shell escaping bug — local Mac shell
   expanded `$MERGED` to empty before the heredoc reached the pod
   (fix: `Write` the script locally + `scp` over instead of inlining);
   (b) `transformers` v5.5 + `torch` 2.10 incompatibility in
   `transformers.integrations.moe._grouped_mm_fallback`'s torch custom
   op (fix: `pip install 'transformers<5'` → 4.57.6).
5. **Attempt 5:** Same script with the downgraded transformers. Merge
   succeeded (15 GB merged fp16 model). Convert step **failed:**
   `ModuleNotFoundError: No module named 'sentencepiece'`. Installed
   it, ran the convert + quantize + upload steps directly (skipping
   the merge since it was cached on disk).

**Final pipeline that worked:**
- `peft.PeftModel.from_pretrained(base_fp16, 'lerugray/hammerstein-7b-lora').merge_and_unload()`
  → 15 GB fp16 HF model
- `python convert_hf_to_gguf.py --outfile v3a.bf16.gguf --outtype bf16` → 14 GB BF16 GGUF
- `llama-quantize v3a.bf16.gguf v3a.Q4_K_M.gguf Q4_K_M` → 4.7 GB Q4_K_M
- `huggingface_hub.HfApi.upload_file(...)` → live on HF

After upload, two GGUFs coexisted (old v1 `Qwen2.5-7B-Instruct.Q4_K_M.gguf`
+ new v3a `hammerstein-7b-lora.Q4_K_M.gguf`). Cleaned up via HF API:
deleted the v1 file + updated the Modelfile's `FROM` directive to
point at v3a.

**Lessons:**
- Unsloth's GGUF push wrapper has too many implicit assumptions
  (system packages, capture_output hiding errors, opaque cleanup on
  failure). For one-shot conversion, the manual pipeline (peft + llama.cpp
  binaries + huggingface_hub) is more legible.
- Container disk for full peft fp16 merge needs ≥60 GB, not 30 GB.
- Shell escaping at heredoc + nested SSH is fragile. Default to writing
  scripts locally and scp'ing them when the script has any variable
  interpolation.
- transformers v5 + torch 2.10 has a known custom-op compatibility
  issue with `transformers.integrations.moe`. Pin transformers <5
  for now.

## What this session means in framework terms

Three layers of the framework auditing itself, again:

1. **The single-variable discipline saved the experiment.** The brief
   proposed a confounded v2 (data + teacher + epochs all changed at
   once). Hammerstein's pre-flight audit said no. Running v2a + v2b
   in parallel cost the same total dollars but produced two clean
   datapoints instead of one muddled one. Without that discipline,
   v3a's design (v2a's data + v2b's lesson about register mismatch
   redirected into off-domain mixin instead of teacher swap) wouldn't
   have been visible.

2. **The "deferred mitigation" pattern worked.** v1 shipped with a
   known limitation explicitly documented in the model card with the
   mitigation named. That documentation made v3a's design obvious
   the moment v2's results landed. The framework's "log negatives
   aggressively" rule converted into a forward-looking design hint
   that paid off the next day.

3. **The blind LLM judge was the load-bearing measurement.** The
   raw marker count and OOD numbers told a clean story, but they're
   form-level. The judge measured actual reasoning quality and
   confirmed v3a's improvement is qualitative, not metric-gaming.
   Running the judge ($0.40, 2 min) was the difference between
   "we have suggestive numbers" and "we have a defensible launch
   swap."

The corpus + framework keep appreciating; the model is the snapshot.
This session shipped v3a as a better snapshot than v1 — same training
recipe template, same cost class (~$2 incremental), measurably
healthier behavior. That's exactly the iteration loop the framework
exists to demonstrate.
