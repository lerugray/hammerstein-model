## Headline outcomes

1. **Wargamer-mode UI shipped end-to-end.** The Claude Design
   port from 2026-05-08-part-3 ([Phase 6.1](2026-05-08-part-3.md))
   was static-fixture-only ŌĆö UI behavior real, "Issue orders"
   stubbed to flip to the streaming-state fixture. This session
   wired it through to `hp_vision.py` via FastAPI, added persistent
   campaign state, and validated the full multimodal pipeline on
   a real CSL game with a real BGG-submitted board photo.

2. **NotebookLM-style Sources panel.** Wargamers can drop a
   rulebook PDF and a strategic-map / counter-sheet image into
   each campaign; both persist with the campaign and ground every
   subsequent issue call. PDFs auto-convert via `pymupdf4llm` +
   the cleanup pipeline ported from
   [`conflict-simulations/manuals/build-docx.py`](https://github.com/lerugray/conflict-simulations/blob/main/manuals/build-docx.py).
   Reference images are attached as image_blocks every turn,
   distinguished from per-turn board photos by a hint in the
   status text.

3. **Karpathy three-layer mini-mini digest layer.** After
   PDFŌåÆMD conversion, an LLM call generates a focused "AI
   Commander Reference" digest (~3k tokens, structured by
   operational concerns: combat resolution, air missions,
   logistics, victory). The preamble injection prefers the
   digest over the full rulebook by default. Empirical: 5x
   reduction in preamble tokens, comparable-or-better order
   quality (the model cites MORE specific rule sections with
   the digest because the structure makes section IDs more
   salient than they were buried in the full narrative).

4. **End-to-end validated on real CSL material.**
   Test campaign: `wargames/2022-ukraine/` with the *2022:
   Ukraine* (CSL) v1.12 PDF as a rules source + CSL box-cover as
   a reference image + the BGG-submitted "post-Turn-1" board photo
   ([BGG image #8416326](https://boardgamegeek.com/image/8416326/2022-ukraine)
   by user `philb1642`) as the per-turn input. Sonnet 4.6 read the
   board correctly (3 axes identified, naval/air boxes spotted,
   turn/VP tracks noted), generated counterfactual Turn 2 orders
   with specific rule references (┬¦3.7.2.1 Railheads, ┬¦3.1.1
   Engineer +1 UQR, ┬¦11.4 Missile Barrage, ┬¦6.5 Ground Support,
   ┬¦7.3 Meeting Engagement, ┬¦11.5 Knockout, ┬¦11.6.2 Partisans,
   etc.), and explicitly distinguished the persisted box-cover
   reference image from the live game-state photo in the
   "What I see on the board" sanity-check belt.

5. **Five rounds of polish + hardening.** After Ray confirmed
   the layout, a wrap-up pass added: reference-image thumbnails
   in the Sources panel (static-file route + `<img>`),
   regenerate-digest button (refresh-icon per rules row),
   `OPENROUTER_API_KEY` precondition check on the issue endpoint
   (clean 503 instead of subprocess error), auto-incrementing
   `tasks.json:current_turn` after each issue, token-budget
   warning at >30k preamble tokens, and `/api/wargame/healthz`
   for upload-flow precondition diagnostics.

6. **Bug found via Playwright self-QA.** Ray reported turn-log
   content was getting cut off. Set up a Playwright headless
   screenshot tool ([`web/backend/_screenshot.py`](../web/backend/_screenshot.py),
   gitignored), captured + measured the DOM. Diagnosis:
   `.wg-turncard` had `overflow: hidden` and was being shrunk
   by its parent flex column ŌĆö each card rendered at ~174px
   when content needed 410-480px, clipping the entire INTENT line
   and most of the status. Fix: `flex-shrink: 0` on the card.
   Also cleaned up two latent bugs caught in the same audit:
   the issue endpoint was persisting the model-prompt-prefix
   ref-note into the turn-log entry's `status` field; and the
   existing test entries had mojibake from earlier `curl -F`
   tests (real React UI input is properly UTF-8, only my
   command-line tests had the issue).

## What this session means in framework terms

The wargamer-mode UI is the **first plausibly-distributable**
applied surface for the framework. Until now the framework's
applied surfaces have been Ray-only (audits over Ray's projects,
self-improving methodology, model training). The wargamer mode
gives a specific community with a concrete pain point a tool
that uses the framework's structural reasoning under the hood
(Auftragstaktik / mission-type orders / decisive-point
identification all map onto the framework's "main effort vs
supporting effort vs reserves" structural primitive).

The Karpathy three-layer pattern (raw ŌåÆ wiki ŌåÆ schema) borrowed
from raybrain is the architectural pattern that makes this
shippable to other people: the rulebook upload flow turns "I
have a copy of this game" into "the AI knows your rules" without
the operator having to hand-curate context. That's what makes it
a **product** rather than a **bespoke setup Ray maintains for
his own play**.
